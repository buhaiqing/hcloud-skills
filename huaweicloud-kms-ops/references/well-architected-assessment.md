# KMS Well-Architected + Three-Pillar Assessment

## 1. Security Pillar (安全支柱)

### IAM Minimum Permissions

| Role | Required Permissions | Use Case |
|---|---|---|
| KMS Viewer | `kms:listKeys`, `kms:describeKey`, `kms:listGrants`, `kms:listKeyRotation` | Read-only inspection |
| KMS Operator | Viewer + `kms:createKey`, `kms:enableKey`, `kms:disableKey`, `kms:scheduleKeyDeletion`, `kms:showKeyPolicy` | Key lifecycle management |
| KMS Admin | Operator + `kms:createGrant`, `kms:revokeGrant`, `kms:importKeyMaterial`, `kms:createDatakey`, `kms:decryptDataKey` | Advanced key operations |

### Key Security Best Practices

| Principle | Rule |
|---|---|
| Default deny | IAM policies grant least-privilege; never use `kms:*` in production |
| Grant cleanup | Revoke all grants before `schedule-key-deletion` |
| Key rotation | Enable automatic rotation for long-lived symmetric keys (365-day interval) |
| Deletion window | Default 30-day window minimum; increase for critical keys |
| BYOK verification | Verify key material fingerprint after import |
| Audit | Tag every CMK with `owner`, `purpose`, `cost-center`, `lifecycle` |

### Encryption Scope

- KMS encrypts **keys**, not data directly.
- Data is encrypted with DEKs generated by KMS.
- DEKs are encrypted (wrapped) by the CMK.

## 2. Stability Pillar (稳定支柱)

### Key Availability

| Practice | Recommendation |
|---|---|
| Single key for all services | Acceptable for dev/test; use per-service keys for prod isolation |
| Key deletion | Always schedule, never force-delete; 30-day minimum window |
| Cross-region key copy | Use key copy via CESM (not natively supported by KMS API) |

### Backup / DR

- KMS keys are **not** backed up by the customer — Huawei Cloud manages HSM replication.
- Key material cannot be exported (except BYOK import).
- DR strategy: replicate key policies (IAM), not key material.

## 3. FinOps (财务运营)

### 3.1 Cost Model

| Item | Cost |
|---|---|
| CMK creation | Free |
| Symmetric key usage | Free (within free tier: 10,000 calls/month) |
| HSM key storage | Billed per key per month after free tier |
| BYOK key material transport | Customer bears external transport cost |
| Key rotation | Free |

### 3.2 Quota Management

| Metric | Default | Unit |
|---|---|---|
| CMK per account per region | 100 | Keys |
| Grants per key | 100 | Grants |
| Aliases per key | 10 | Aliases |

### 3.3 Idle Key Detection

```bash
# Keys that exist but have not been used in 90 days
hcloud kms list-keys --region {{env.HW_REGION_ID}} \
  | jq '.keys[] | select(.key_state == "ENABLED")'
# Cross-reference with KMS API call metrics in CES
```

> Idle CMKs still consume quota. Disable or schedule deletion if unused.

### 3.4 Right-Sizing

| Observed | Recommendation |
|---|---|
| > 50 CMKs, most < 5% utilization | Consolidate to shared CMK for non-production |
| BYOK in use, high transport cost | Evaluate CMK vs BYOK cost trade-off |
| All keys with rotation ON | Evaluate rotation need — rotation costs nothing but adds key versions |

## 4. SecOps (安全运营)

### 4.1 Grant Attack Surface

| Risk | Mitigation |
|---|---|
| Grant proliferation | Max 100 grants per key; audit monthly |
| Stale grants | Revoke grants when grantee service is decommissioned |
| Over-privileged grants | Use `operations` field to limit grant to specific ops only |

### 4.2 Deletion Cascade

| Risk | Mitigation |
|---|---|
| Data loss after deletion window | Never delete a key with active dependent resources |
| OBS/RDS encrypted with deleted key | Confirm no OBS buckets / RDS instances depend on the CMK |
| Grant remaining at deletion | Always `list-grants` and `revoke-grant` before scheduling deletion |

### 4.3 Threat Detection Triggers

| Trigger | Action |
|---|---|
| Key disabled without user action | Audit IAM logs via `huaweicloud-cts-ops`; check for unauthorized `disable-key` |
| High API call volume spike | Delegate to `huaweicloud-ces-ops`; check for compromise |
| Grant count approaching 100 | Alert via `huaweicloud-ces-ops` |

## 5. Operational Efficiency

- Use aliases for human-readable key identification.
- Tag every CMK: `owner`, `purpose`, `cost-center`, `lifecycle=permanent|ephemeral`.
- Automate grant revocation on service decommission.
- Enable CES alarms on key state changes.
